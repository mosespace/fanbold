# Contributing

Thank you for your interest in contributing to Fanbold! We appreciate your willingness to help make this tool better for everyone. Your help is essential for keeping it great.
Contributions to this project are released to the public under the project's open source license.

Please note that this project is released with a Contributor Code of Conduct. By participating in this project you agree to abide by its terms.

## Code of Conduct

All contributors are expected to abide by our project's [Code of Conduct](CODE_OF_CONDUCT.md). Please ensure you have read and agree to its terms before participating.

---

## How Can I Contribute?

There are many ways to contribute to Fanbold:

1.  **Search Existing Issues:** Check the Issue Tracker to see if the bug has already been reported.
2.  If it's a new bug, please open a new issue and include clear steps to reproduce the issue and details about your environment (browser, OS).
3.  **Submit a Feature Request:** See if your idea has already been suggested.If not, open a new issue and clearly describe the problem your idea solves and how it might look in the interface.

## ðŸ’» Local Development Setup

To run Fanbold locally and start contributing code, follow these steps:

1. You must have **Git** and **Node.js** (including npm or yarn) installed on your system.
   2.First, **Fork** the Fanbold repository via the GitHub interface. Then, clone your fork to your local machine:

```bash
# Clone your fork
git clone [https://github.com/YOUR_USERNAME/fanbold.git](https://github.com/YOUR_USERNAME/fanbold.git)
cd fanbold
```

3. Install the project dependencies using your package manager:

```bash
# Using npm
npm install

# OR using yarn
 yarn install
```

4. Run the Dev Server

```bash
npm run dev
# OR yarn dev
```

The application should now be available locally (usually at http://localhost:3000).

5. Create a new branch: git checkout -b my-branch-name
6. Make your changes and test them locally.
7. Push to your fork and submit a pull request
   In your Pull Request description, ensure you check off these items:
   + The code follows the project's existing style guidelines.
   + I have tested my changes locally.
   + My commit message adheres to the Conventional Commits standard.
   + The PR title clearly summarizes the change and references the related issue (e.g., (Closes #12)).
8. Pat your self on the back and wait for your pull request to be reviewed and merged.
